IP Overview
-----------

A byte-sliced arithmetic logic unit (ALU) for a RISC-V RV32E core. The ALU
operates on 8-bit slices of 32-bit words, processing one slice per clock cycle
over four cycles to produce a full 32-bit result. Carry state is propagated
between slices via an internal flip-flop.

Design Constraints
------------------

[cols="1,4"]
|===
| Constraint ID | Description

| `DC-ALU-010`
| `XLEN` *MUST* be a positive integer multiple of `SLICE_WIDTH`.

| `DC-ALU-020`
| The caller *MUST* present slices in order from LSB (``count_i`` = 0) to MSB
(``count_i`` = `SLICE_ADDR_WIDTH - 1`) on consecutive rising clock edges.

| `DC-ALU-030`
| `alu_op_i` *MUST NOT* change between slices of the same operation.
|===

Design Requirements
-------------------

[cols="1,4"]
|===
| Requirement ID | Description

| `DR-ALU-010`
| The module *MUST* implement ADD: `result = a + b`.

| `DR-ALU-020`
| The module *MUST* implement SUB: `result = a - b`.

| `DR-ALU-030`
| Carry *MUST* be propagated correctly from each slice to the next.

| `DR-ALU-040`
| On the first slice (`count_i == 0`), the carry-in *MUST* be initialised to
`negate_b` to provide the two's complement +1 for subtraction.

| `DR-ALU-050`
| On subsequent slices (`count_i != 0`), the carry-in *MUST* be the carry-out
stored from the previous slice.

| `DR-ALU-060`
| The carry flip-flop *MUST* be reset to zero when `rst_ni` is deasserted.

| `DR-ALU-070`
| For ADD, `carry_o` on the final slice (`count_i == SLICE_ADDR_WIDTH-1`) *SHALL*
be 1 if the unsigned 32-bit result overflows, and 0 otherwise.

| `DR-ALU-080`
| For SUB, `carry_o` on the final slice *SHALL* be 1 if no borrow occurred
(a >= b), and 0 if a borrow occurred (a < b). This is the natural carry-out
of two's complement subtraction and is the inverse of the borrow signal.

Architecture
------------

The ALU uses a byte-sliced datapath inspired by classic bit-sliced CPU
architectures. A full 32-bit operation is broken into four 8-bit slice
operations, one per clock cycle, with `count_i` indicating the current slice.

=== Negation and Subtraction

Subtraction is implemented using two's complement negation, sharing the same
adder hardware as addition. A `negate_b` signal is derived combinationally
from `alu_op_i` using a Karnaugh map:

[source]
----
negate_b = alu_op_i[3] || (!alu_op_i[2] && alu_op_i[1])
----

When `negate_b` is asserted, `b_i` is inverted via XOR before entering the
adder, and the carry-in on the first slice is set to 1, completing the two's
complement negation (`~b + 1`).

=== Adder

The adder is 9 bits wide to capture carry-out:

[source]
----
adder_a = {1'b0, a_i}
adder_b = {1'b0, (b_i ^ {SLICE_WIDTH{negate_b}})}
adder_result = adder_a + adder_b + carry
----

The result slice is `adder_result[SLICE_WIDTH-1:0]` and the carry-out is
`adder_result[SLICE_WIDTH]`.

=== Carry Propagation

The carry-in for each cycle is selected as:

[stem]
++++
\texttt{carry} =
\begin{cases}
  \texttt{negate\_b} & \text{if } \texttt{count\_i} = 0, \\
  \texttt{carry\_q}  & \text{otherwise.}
\end{cases}
++++

`carry_q` is a registered copy of `carry_d` (the carry-out from the previous
slice), updated on each rising clock edge.

=== Implementation Notes

* The `negate_b` Karnaugh map covers ADD, SUB, SLT, and SLTU. If `fu_op_t`
  encoding changes, this logic *MUST* be re-derived.

* The adder is shared between ADD, SUB, SLT, and SLTU (all `alu_op_i[2:0]`
  matching `3'b0??`), which is possible because their lower three bits are
  identical and the distinction is handled entirely by `negate_b`.
|===
